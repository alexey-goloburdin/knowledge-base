E2E-тесты, кто не в курсе, проверяют работу всего приложения в сборке. Если мы говорим о вебе, то E2E-тесты ведут себя как обычные пользователи. Открывают браузер, в нём нашу систему и протыкивают логику. Вызываются все слои: фронтенд, бэкенд, базы данных, очереди, всё. Можно протыкивать даже реальную систему, в которой работают пользователи. Ну, или развернуть тестовый стенд, идентичный реальному продакшну.

С точки зрения выявления багов, то есть ошибок, это наиболее полезные тесты, потому что тестируют всё в сборке.

Итак, что нужно модели для взаимодействия с браузером? Ей нужен внешний инструмент. Он будет выполнять команды в браузере и возвращать результаты в модель. И нужен какой-то агент, который будет этим управлять. Агентом может быть, например, ваш код или готовый агент Claude Code (==покажи==), который отлично умеет работать с локальными LLM, но нужен аккаунт Anthropic для начальной аутентификации, или агентом может быть маленький минималистичный опенсорсный агент pi (==покажи==), который я сейчас покажу.

## Старые наброски по теме

Run Playwright MCP server on Windows in powershell:

```powershell
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser
npx @playwright/mcp@latest --port 3000 --host 0.0.0.0 --allowed-hosts '*'
```

Check IP:

```powershell
ipconfig
```

Get WSL address, for example, `172.27.112.1`.

`.mcp.json` for pi:

```json
{
  "mcpServers": {
    "playwright": {
      "url": "http://172.27.112.1:3000/mcp"
    }
  }
}
```

Run pi:

```shell
http_proxy="" https_proxy="" pi
```
